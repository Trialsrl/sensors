Pronto. Inserisci i valori, tocca “Scrivi su tag”, appoggia il telefono sul chip e tienilo fermo finché non compare “Scritto”.
Un tag nuovo o vuoto va inizializzato una volta con l'app ST25 (Initialize NDEF). Web NFC non sa creare la struttura NDEF da zero sui tag ISO 15693: dopo l'init, scrittura e lettura da qui funzionano.
Per il firmware — dove leggere via I2C { }

La memoria utente è leggibile in I2C a partire da 0x0000. I primi byte sono il Capability Container: 4 byte sul 04KC, 8 byte su 16KC e 64KC. Questi 4 byte di differenza spostano tutto ciò che segue, quindi l'indirizzo assoluto del payload cambia solo tra il 04KC e gli altri due. Il record scritto da questa app è un record NDEF di tipo unknown (TNF 0x05, senza campo type): cortissimo, così la scrittura è quasi istantanea e difficile da interrompere.

Il modo robusto, valido per entrambi i chip, è non fissare l'indirizzo ma fare il parsing del TLV: il parser sotto rileva da solo la lunghezza del CC dal byte 2 (0x00 → CC di 8 byte). Conferma una volta con un dump I2C: dopo il CC deve esserci 0x03.

// Legge i 6 byte di payload dal tag — funziona su 04KC, 16KC e 64KC.
// buf = memoria utente letta da I2C a partire da 0x0000.
int read_config(const uint8_t *buf, uint16_t *ppr,
                uint16_t *limA, uint16_t *limB) {
  uint16_t i = (buf[2] == 0x00) ? 8 : 4;  // lunghezza CC: 8B (64KC) o 4B (04KC)
  while (i < 32 && buf[i] == 0x00) i++;     // salta eventuali NULL-TLV
  if (buf[i] != 0x03) return -1;          // nessun messaggio NDEF
  i++;                                       // punta al campo lunghezza
  i += (buf[i] == 0xFF) ? 3 : 1;          // L su 1 byte, o 3 byte se 0xFF
  uint8_t flags = buf[i], typeLen = buf[i+1];
  uint16_t p = i + ((flags & 0x10) ? 3 : 6) + typeLen; // SR → payload len su 1 byte
  *ppr  = (buf[p+0] << 8) | buf[p+1];   // big-endian
  *limA = (buf[p+2] << 8) | buf[p+3];
  *limB = (buf[p+4] << 8) | buf[p+5];
  return 0;
}
